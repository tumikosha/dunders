# Tree-дандлер с редактируемыми сниппет-узлами — дизайн

Дата: 2026-06-15
Статус: утверждён (дизайн), готов к плану реализации

## Цель

Новый «дандлер» (`WindowContent`) проекта `dunders`, отображающий **дерево**,
у которого узел может разворачиваться в **редактируемый текстовый сниппет**
(несколько строк), редактируемый уже существующим редактором проекта
(`EditorWidget`) — без бордюров/декораций окна, инлайн внутри дерева.

Каркас задуман как **универсальный, app-agnostic строительный блок**: узлы и
редактируемые сниппеты — это просто примитивы, а конкретное наполнение
(файловая система, JSON, аутлайнер, …) приходит позже от отдельных адаптеров.
Проверочные сценарии, под которые модель обязана подходить: дерево ФС
(директории-ветки + файлы-листы с редактируемым содержимым) и просмотр JSON
(объекты/массивы-ветки + скаляры-листы с редактируемым значением).

## Принятые решения (из брейншторма)

1. **Назначение** — универсальный переиспользуемый каркас; конкретные адаптеры
   делаются позже отдельно.
2. **Модель узла** — «заголовок + опциональное тело + опциональные дети».
   Формальное надмножество, покрывающее и ФС, и JSON: заголовок (имя файла /
   ключ) есть всегда; «дети» и «тело» в этих сценариях взаимоисключающие, но
   модель допускает и то и другое одновременно (запас прочности для аутлайнера
   и т.п.).
3. **Редактор в узле** — **один активный редактор**. Раскрытые тела рисуются как
   статичный многострочный текст с подсветкой; по входу в правку ровно один
   `EditorWidget` монтируется на место тела активного узла, по выходу текст
   сохраняется обратно в модель. Масштабируется на тысячи узлов.
4. **Источник данных** — **модель в памяти + ленивая догрузка детей** через
   `callback` при первом раскрытии (нужно для большой ФС). Правка тела меняет
   объект узла и испускает событие «тело изменилось»; куда сохранять — дело
   владельца/адаптера, каркас сам ничего не персистит.
5. **Скоуп итерации** — каркас (виджет + модель + встроенный редактор +
   навигация) + небольшой in-memory демо-наполнитель в `windowing/demo` + юнит- и
   smoke-тесты. Боевые адаптеры (ФС/JSON) — за рамками.
6. **Групповая селекция** — каркас ведёт **набор выделенных узлов** (отдельно от
   курсора и от режима правки), подсвечивает их и постит `SelectionChanged`.
   Конкретные действия над группой — дело владельца/адаптера (позже). Как
   multi-select у `FilePanel`.

## Расположение и слои

App-agnostic, по конвенции проекта («`windowing/core/` — editor-agnostic
примитивы»; submodules наружу не торчат, публичный API реэкспортируется из
`windowing/__init__.py`):

- `windowing/core/tree_model.py` — **чистая** модель и логика, без Textual
  (рядом с `buffer.py`, `fold_engine.py`). Полностью юнит-тестируемая.
- `windowing/tree/widget.py` — `TreeViewWidget(ScrollView)`: отрисовка,
  навигация, встроенный редактор.
- `windowing/tree/content.py` — `TreeContent(WindowContent)`: «дандлер»
  (заголовок окна, `get_commands()`, dirty-маркер).
- Реэкспорт `TreeContent`, `TreeNode` из `windowing/__init__.py`.

## Модель узла (`TreeNode`)

```python
TreeNode:
    label: str                       # строка в дереве (имя файла / ключ JSON)
    body: str | None = None          # редактируемый сниппет; None = тела нет
    children: list[TreeNode] = []
    expanded: bool = False           # раскрыты ли дети
    body_open: bool = False          # показано ли тело под заголовком
    loader: Callable[[TreeNode], list[TreeNode]] | None = None  # ленивая догрузка
    loaded: bool = False
    parent: TreeNode | None = None
    data: Any = None                 # произвольная нагрузка адаптера (путь, json-ref…)
```

Чистые, тестируемые методы:
- `add_child(child)` — добавить ребёнка, проставить `parent`.
- `is_branch` — у узла есть дети или `loader` (т.е. он раскрываемая ветка).
- `ensure_loaded()` — если есть `loader` и ещё не `loaded`, вызвать его один раз,
  записать детей, выставить `loaded = True`. Идемпотентно.

## Мост модель → экран

`flatten_visible(root) -> list[Row]` — чистая функция, обходит модель с учётом
`expanded`/`body_open` и возвращает плоский список визуальных строк двух видов:

- **строка-заголовок** — несёт ссылку на `TreeNode`, глубину, признак
  ветки/раскрытия;
- **строка-тела** — несёт ссылку на `TreeNode` и индекс строки внутри сниппета.

Это и тестируется отдельно, и кормит рендер виджета. Без Textual.

## Виджет (`TreeViewWidget`, `ScrollView`)

Рисует строки сам в `Strip`'ы (как `EditorWidget`/`FilePanel`): отступы-гайды,
стрелка `▸`/`▾` для веток, маркер пометки (`*`/подсветка) у выделенных узлов,
`label`, и статичные строки тела с подсветкой (переиспользуем `core/highlight`).

Три **независимых** состояния:
- «текущая строка» — курсор навигации (одна, на заголовках узлов);
- набор помеченных узлов (`selected: set[TreeNode]`) для групповой селекции;
- режим правки (смонтирован один `EditorWidget`).

**Один активный редактор.** Жизненный цикл правки:

1. Вход в правку узла `n` (тело есть): `buf = TextBuffer.from_string(n.body or "")`,
   монтируется один `EditorWidget(buffer=buf)` на место тела активного узла,
   получает фокус.
2. Выход (`Esc` / коммит): `new = "\n".join(buf.lines)`. Если `new != n.body` —
   `n.body = new`, постится `TreeViewWidget.BodyEdited(node=n)` и
   `is_dirty = True`. Редактор размонтируется, фокус возвращается на дерево.

В каждый момент времени смонтирован максимум один `EditorWidget`.

### Технический риск и фолбэк

Главный риск — позиционирование инлайн-редактора внутри прокрутки: редактор
должен «ехать» вместе со скроллом тела. Митигация: при входе в правку держим
активный узел видимым и пересчитываем offset на события скролла.

**Утверждённый фолбэк:** если инлайн-оверлей окажется слишком капризным —
правка ведётся в зафиксированной строке-доке снизу контента (тот же
`EditorWidget`, без бордюров), без потери модели и контракта `BodyEdited`.
Выбор инлайн vs док решается эмпирически на этапе реализации.

## Навигация

Идиома mc / Turbo Vision.

Клавиатура — навигация:
- `↑` / `↓` — перемещение курсора по заголовкам узлов;
- `→` — раскрыть ветку / открыть тело листа; `←` — свернуть / перейти к родителю;
- `Enter` / `F4` на узле с телом — открыть тело и войти в правку;
- `Esc` — закоммитить правку, размонтировать редактор, вернуться в навигацию;
- `Space` — переключить раскрытие.

Клавиатура — групповая селекция (mc-идиома, без конфликта со `Space`):
- `Insert` — пометить/снять пометку текущего узла и сместить курсор вниз;
- `Shift+↑` / `Shift+↓` — диапазонная пометка.

Мышь:
- колесо — скролл дерева (даёт `ScrollView`);
- клик по стрелке `▸`/`▾` — раскрыть/свернуть ветку;
- клик по заголовку — выделить узел (курсор) и **сбросить** набор пометок;
- клик по сниппет-узлу (заголовку с телом или по строке тела) — войти в правку и
  поставить курсор в точку клика;
- `Ctrl+клик` — добавить/убрать узел из набора пометок;
- `Shift+клик` — диапазонная пометка от текущего курсора до кликнутого узла.

## Контракт правок и хранение

Каркас **не сохраняет** ничего сам: держит модель в памяти и постит события:
- `BodyEdited(node)` — тело сниппета изменилось;
- `SelectionChanged(nodes)` — изменился набор помеченных узлов.

Куда писать (файл, JSON, …) и что делать с группой — дело владельца/адаптера.
Ленивые дети — через `node.loader`, вызываемый один раз при первом раскрытии.

## `TreeContent` (дандлер)

`TreeContent(WindowContent)` оборачивает `TreeViewWidget`:
- `window_title` — задаётся при создании;
- `get_commands()` — focus-scoped команды: Expand/Collapse, Edit snippet (`F4`),
  Commit (`Esc`), Toggle mark (`Insert`); состав уточняется при реализации, но без
  add/delete-узлов (YAGNI, на потом);
- проксирует `BufferModified`/`BodyEdited` в `is_dirty` окна.

## Демо и тесты (скоуп итерации)

Демо:
- in-memory наполнитель строит образец дерева: пара папок, листы-сниппеты и одна
  **ленивая** ветка (демонстрирует `loader`);
- регистрируется новым окном в `windowing/demo/app.py`; запуск
  `python -m dunders.windowing.demo`.

Тесты:
- юнит-тесты `tree_model`: expand/collapse; ленивая догрузка (loader зовётся
  ровно один раз); `flatten_visible` для разных состояний; write-back тела;
  логика набора пометок (toggle, диапазон, сброс);
- async smoke-тест: монтирование `TreeContent`, цикл «войти в правку → изменить
  → Esc → проверить `node.body` и `is_dirty`»; пометка узлов (`Insert`/диапазон) →
  проверить `SelectionChanged` и набор.

## Явно вне скоупа (YAGNI)

- Боевые адаптеры ФС/JSON (отдельная итерация).
- Добавление/удаление/перемещение узлов, drag-and-drop.
- Несколько одновременно живых редакторов.
- Персистентность (только событие `BodyEdited`).
- Конкретные действия над помеченной группой (только набор + `SelectionChanged`).
